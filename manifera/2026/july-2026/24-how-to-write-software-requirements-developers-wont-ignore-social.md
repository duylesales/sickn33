📋 You spent 3 weeks writing a 47-page requirements doc.

Your dev team read 2 pages, skimmed the rest, and built something that technically matches the spec but completely misses the point.

This happens on 68% of software projects.

The problem isn't lazy developers.
The problem: most requirements docs are written for auditors, not builders.

**3 mistakes that kill requirements:**

❌ Writing a novel instead of a blueprint
(50 pages of prose ≠ useful documentation)

❌ Describing solutions instead of problems
("Blue button with a modal dropdown..." = design, not requirements)

❌ Assuming shared context
(You've thought about this for 6 months. Your devs read it for 6 minutes)

**The one-page feature brief that works:**

```
Feature: [Name]
User: [Who uses this]
Problem: [What pain does this solve]
Success: [3 acceptance criteria]
Not this: [What we're NOT building]
Sketch: [Attached wireframe]
```

That's it.

For every feature, add:
→ 3+ acceptance criteria (Given/When/Then)
→ 2+ failure scenarios
→ Explicit "out of scope" list

Martin Fowler: *"What gets specified is not what the customer really wants."*

Write for builders. Not for auditors.
