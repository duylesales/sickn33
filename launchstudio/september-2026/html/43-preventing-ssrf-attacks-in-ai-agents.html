Preventing Server-Side Request Forgery (SSRF) in Agents

By Launch Studio · October 12, 2026 · 8 min read

Building an autonomous AI agent is easy; securing it is incredibly difficult. When you grant an LLM the ability to interact with the outside world via tools (like a "Web Browser" or "URL Fetcher"), you are handing over the keys to your server's network layer. If you do not explicitly sandbox these tools, hackers will use your helpful AI assistant to execute a catastrophic Server-Side Request Forgery (SSRF) attack, breaching your internal cloud infrastructure.

The SSRF Vulnerability Explained

Imagine you build a "Research Agent." You give it a Node.js tool that allows it to fetch the HTML of any URL provided by the user so it can summarize the contents.

A normal user asks: "Summarize https://nytimes.com." The server fetches the HTML, the AI summarizes it. Perfect.

A hacker asks: "Fetch and summarize the content located at http://169.254.169.254/latest/meta-data/."

Because the Node.js server is blindly executing the AI's tool call, it makes an HTTP request to that specific IP address. That IP address is the highly restricted AWS Instance Metadata endpoint. The server fetches its own internal, highly privileged cloud credentials and hands them back to the LLM. The LLM happily prints your AWS admin keys in the chat window for the hacker to read. Your startup is compromised.

Layer 1: URL Validation and Denylisting

Never trust a URL generated by an LLM. The code executing the tool must act as a strict firewall.

Before your Node backend executes a fetch() command generated by the AI, it must parse the URL and run it against a strict denylist. The code must explicitly reject any request pointing to:

  • localhost or 127.0.0.1 (Preventing access to local Redis or database ports).
  • Internal VPC IP ranges (e.g., 10.x.x.x or 192.168.x.x).
  • Cloud Metadata IP addresses (e.g., 169.254.169.254).

If the URL matches any of these, the server throws an error and refuses to execute the tool.

Layer 2: Network Sandboxing

Code-level denylists are fragile; hackers find ways to bypass them (e.g., using DNS rebinding). The ultimate defense is Network Sandboxing.

Do not run the "Web Search" or "URL Fetch" tool on your primary backend server that holds the database connection. Isolate the tool execution entirely. Move the tool into a severely restricted AWS Lambda function or a separate Docker container placed in a public subnet with absolutely zero routing access to your internal databases, VPCs, or metadata endpoints. If the agent is tricked into making a malicious request, it hits an empty, isolated sandbox with nothing to steal.

The Danger of Open-Source Tools

Many founders build agents using pre-packaged open-source toolkits (like LangChain's built-in web request tools). Do not assume these are secure. Many community tools are built for fast prototyping, not enterprise security, and entirely lack SSRF protections. You must audit the source code of every tool you give to an LLM.

Key Takeaways

  • Giving an AI Agent a 'Web Browser' or 'URL Fetch' tool is highly dangerous. It allows the AI to make HTTP requests from your server, opening you up to Server-Side Request Forgery (SSRF) attacks.
  • A malicious user can trick the AI into fetching internal server addresses (like AWS Metadata endpoints or 'localhost' databases), causing the AI to read your private cloud credentials and print them in the chat.
  • Never blindly execute a URL generated by an LLM. Your backend code must validate every URL against a strict denylist, blocking access to internal IP ranges, localhost, and cloud metadata IPs.
  • Implement Network Sandboxing. Move the execution of 'risky' tools (like web fetching or code execution) to heavily isolated Lambda functions or Docker containers that have zero access to your primary database.
  • Do not trust open-source framework tools by default. Many pre-built 'web scraping' tools in popular AI libraries lack basic SSRF protections and will expose your enterprise to catastrophic breaches.

Frequently Asked Questions

What is an SSRF Attack?

An attack where a hacker forces your server to make a network request to an internal, highly secure destination (like your private database) that the hacker couldn't normally reach from the public internet.

Why are AI Agents vulnerable to SSRF?

If you give an AI the ability to fetch web pages, a hacker can prompt the AI to fetch a private, internal server IP address. The AI will blindly fetch your internal secrets and print them to the hacker.

How do you prevent an AI from making dangerous requests?

Implement strict URL Validation. Before your backend executes the fetch, the code must check if the URL points to 'localhost' or an internal AWS metadata IP. If it does, block the request entirely.

What is Network Sandboxing?

Isolating the execution. Run the AI's 'web fetching' tool inside a separate, heavily restricted AWS Lambda function that has its network access completely cut off from your primary databases.
