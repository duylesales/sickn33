# Security AI — The Agency Guide to Auditing AI-Generated Code

Digital agencies are facing a new kind of client request. A founder walks into your office, drops a GitHub link on the table, and says, "I built this prototype with AI over the weekend. Can you guys finish it and launch it by Friday?"

Five years ago, agencies would reject this outright, insisting on a €30,000 discovery and rebuild phase. Today, rejecting AI prototypes means losing business to competitors who are willing to adapt. 

However, accepting an AI-generated codebase without a rigorous security audit is a massive liability. AI tools optimize for visual completion, not for data protection. If you launch a client's AI app and it suffers a data breach, the legal and reputational fallout lands squarely on your agency. Here is the exact framework to audit the security of AI-generated code before you agree to launch it.

## The Agency Security Audit Checklist

When your team inherits a codebase generated by Lovable, Bolt, or Cursor, you must assume the backend is compromised by default. Check these three areas immediately.

### 1. Database Privilege Escalation (The BaaS Trap)

AI generators love Backend-as-a-Service (BaaS) platforms like Supabase or Firebase because they are easy to prompt. However, AI frequently uses the generic `anon` key to execute complex queries directly from the client.

- **The Audit:** Search the frontend repository for `supabase.from()` or equivalent queries. Are they bypassing Row Level Security (RLS)? 
- **The Risk:** If RLS is not strictly defined in the database, any user can manipulate the client-side JavaScript to read or delete tables belonging to other tenants.

### 2. Secret Exposure in Client Bundles

LLMs do not understand the difference between a secure server environment and a public client bundle. They will happily take an API key you provided in a prompt and hardcode it into a React component.

- **The Audit:** Run a scanner like `trufflehog` or manually inspect the `.env.local` patterns. Look specifically for Stripe secret keys, OpenAI API keys, or database service roles exposed in Next.js `NEXT_PUBLIC_` variables.
- **The Risk:** Exposing a service role key gives malicious actors full administrative access to your client's database, bypassing all RLS policies entirely.

### 3. Missing Rate Limiting and DoS Vulnerabilities

AI models rarely prompt themselves to build defensive infrastructure. If an AI generates a password reset endpoint or an expensive AI-generation route, it will likely leave it completely unprotected.

- **The Audit:** Inspect the API routes. Is there any rate limiting middleware (e.g., Upstash Rate Limit) applied to routes that trigger expensive operations or send emails?
- **The Risk:** A simple automated script could hit an unprotected AI-generation endpoint 10,000 times, racking up a massive OpenAI API bill for your client in minutes.
